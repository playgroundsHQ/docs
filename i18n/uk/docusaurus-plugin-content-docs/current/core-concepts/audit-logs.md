# Журнал аудиту платформи

Журнал аудиту забезпечує **постійну шкалу подій** життєвого циклу на платформі fibe.gg — включаючи Playground, Генії, профілі та серверні ресурси.

## Огляд

Значущі події автоматично записуються як записи журналу аудиту. Ці записи допомагають вам:

- **Діагностувати** — Зрозуміти, що і коли сталося під час життєвого циклу Playground
- **Відстежувати активність** — Бачити, які дії були ініційовані вами, а які — системою
- **Переглядати історію** — Записи зберігаються навіть після видалення Playground

## Де переглядати

Журнали аудиту доступні в кількох місцях залежно від контексту ресурсу:

1. **Сторінка деталей Playground** — Натисніть на бейдж **"Updated"** у заголовку Playground, щоб побачити шкалу останніх подій для цього конкретного Playground.
2. **Перегляд ресурсів** — Перевірте контекстне меню або вкладку історії для конкретних Генії, шаблонів або Playroom.
3. **Сторінка журналу аудиту** — Перейдіть до **Audit Logs** на бічній панелі, щоб побачити пагінований список усіх подій платформи.

### Журнали аудиту Playground

Ви можете відстежувати зміни конкретного Playground безпосередньо у вашій публічній шкалі профілю. Це вмикається в налаштуваннях Генія шляхом вибору `build_in_public_playground_id`. Після налаштування кожна зміна файлу або оновлення сервісу в цьому Playground транслюється вашим публічним підписникам.

## Типи акторів

Кожен запис журналу фіксує, **хто** ініціював дію:

| Актор | Опис |
|-------|------|
| **Player** | Дія, виконана вами вручну (напр., створення, перестворення, оновлення) |
| **System** | Автоматична дія платформи (напр., доставка вебхуків, ініціалізація системи) |
| **Playguard** | Фоновий процес узгодження (напр., лікування дрифту, очищення після закінчення терміну дії) |
| **Genie** | Дія, виконана AI Генієм від вашого імені через MCP-інструменти |
| **API Key** | Дія, ініційована програмно за допомогою обмеженого API-ключа |

## Категорії подій

Події організовані за категоріями:

### Дії гравця

| Подія | Опис |
|-------|------|
| Player created playground | Ви створили новий Playground |
| Player deleted playground | Ви видалили Playground |
| Player triggered rollout | Ви виконали розгортання працюючого Playground |
| Player triggered hard restart | Ви виконали жорсткий перезапуск працюючого Playground |
| Player extended expiration | Ви продовжили TTL |
| Player retried creation | Ви повторили невдале створення |
| Player committed changes | Ви зафіксували код у робочій директорії Playground |

### Життєвий цикл створення

| Подія | Опис |
|-------|------|
| Creation started | Розпочато конвеєр підготовки |
| Creation completed | Усі контейнери запущені та працюють |
| Creation failed | Крок підготовки завершився невдало |

### Дрифт і лікування

| Подія | Опис |
|-------|------|
| Drift detected | Playguard виявив розбіжність між очікуваним та фактичним станом |
| System rollout (drift) | Playguard автоматично виконав розгортання Playground для лікування дрифту |
| Rollout skipped (dirty) | Playguard пропустив розгортання через виявлені незафіксовані зміни |
| Build drift detected | Сервіс у режимі Production має нові коміти, що потребують перебудови |

### Закінчення терміну дії

| Подія | Опис |
|-------|------|
| Playground expired | TTL минув і Playground було очищено |
| Expiration extended | TTL було продовжено |
| Expiration skipped (dirty) | Очищення пропущено через виявлені незафіксовані зміни |

### Системні події

| Подія | Опис |
|-------|------|
| Status changed | Playground перейшов у новий стан |
| Error occurred | Зафіксовано помилку |
| Recovered from interruption | Playground відновився після перезапуску платформи |
| Job completed | Playground у режимі Job завершив виконання |

## Збереження

Записи журналу аудиту **ніколи не видаляються** при знищенні Playground. Назва Playground зберігається безпосередньо в кожному записі журналу, тому повна історія подій залишається доступною навіть після видалення самого Playground.
